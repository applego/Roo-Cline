## 2. Web アプリケーション版 Roo-Cline のアーキテクチャ

### 2.1 VS Code 拡張機能版からの変更点

VS Code 拡張機能版 Roo-Cline は、VS Code の API に深く統合されたアーキテクチャでしたが、Web アプリケーション版では、より汎用的な Web 技術に基づいたアーキテクチャへと再構築されました。

#### 2.1.1 アーキテクチャ全体の概要

Web アプリケーション版 Roo-Cline は、主に以下の要素で構成されます。

- **フロントエンド:** ユーザーインターフェースを提供する Web アプリケーション。React をベースに構築され、ユーザーからの入力を受け付け、バックエンドとの通信を行い、結果を表示します。
- **バックエンド:** エージェントのコアロジック、API 連携、データ処理などを担うサーバーサイドアプリケーション。Node.js (または Python など) で構築され、フロントエンドからのリクエストを処理し、必要なツールや外部 API を利用してタスクを実行します。
- **データベース:** タスク履歴、ユーザー設定、その他必要なデータを永続化するためのデータベース。PostgreSQL, MySQL, MongoDB など、適切なデータベースを選定します。

#### 2.1.2 主要コンポーネントとその役割

- **API Gateway:** フロントエンドからのリクエストをバックエンドの各サービスにルーティングします。
- **Task Manager:** ユーザーからのタスクを受け付け、タスクの実行計画を立て、各コンポーネントにタスクを分配・管理します。
- **Agent Core:** エージェントの主要なビジネスロジックを実装します。プロンプト生成、LLM API 呼び出し、ツール実行、結果解析などを行います。
- **Tool Executor:** LLM からの指示に基づき、様々なツール（コード実行、ファイル操作、Web ブラウジングなど）を実行します。
- **Data Store:** タスクの状態、履歴、設定などを永続化します。

#### 2.1.3 VS Code 依存からの脱却と Web アプリケーションとしての独立

Web アプリケーション版では、以下の点で VS Code への依存を解消し、独立したアプリケーションとして動作します。

- **UI:** VS Code の Webview API ではなく、標準的な Web ブラウザ上で動作する UI を採用します。
- **ファイルシステムアクセス:** VS Code の API を介したファイルシステムアクセスではなく、バックエンドサーバーを介したファイル操作を行います。
- **コマンド実行:** VS Code のターミナル API ではなく、バックエンドサーバー上で直接コマンドを実行する方式を採用します。

### 2.2 技術スタック

Web アプリケーション版 Roo-Cline は、以下の技術スタックをベースに開発されています。

- **フロントエンド:**
    - 言語: TypeScript
    - フレームワーク: React
    - UI ライブラリ: Material UI (または Ant Design, Chakra UI など)
    - 状態管理: Context API, Zustand, Recoil など
    - ビルドツール: Vite, Webpack など
- **バックエンド:**
    - 言語: Node.js (または Python など)
    - フレームワーク: Express.js, NestJS (Node.js の場合), Flask, Django (Python の場合)
    - API クライアント: axios, node-fetch (フロントエンド、バックエンド共通)
    - データベース: PostgreSQL, MySQL, MongoDB など (プロジェクトの要件に応じて選定)
    - ORM/ODM: Prisma, TypeORM (Node.js の場合), SQLAlchemy, Django ORM (Python の場合)
    - テストフレームワーク: Jest, Mocha, Chai (Node.js の場合), pytest, unittest (Python の場合)
- **インフラ:**
    - クラウドプラットフォーム: AWS, Google Cloud, Azure など (またはオンプレミス)
    - コンテナ技術: Docker, Kubernetes (必要に応じて)
    - CI/CD: GitHub Actions, CircleCI, Jenkins など

## 3. ワークフローの変更点

### 3.1 タスク実行フローの概要 (Web アプリケーション版)

Web アプリケーション版 Roo-Cline におけるタスク実行の基本的なフローは以下の通りです。

1.  **ユーザーからのタスクリクエスト:** ユーザーは Web ブラウザ上の UI を通じてタスクを Roo-Cline にリクエストします。
2.  **リクエストの受付とタスクの生成:** フロントエンドはユーザーのリクエストを API Gateway を経由してバックエンドに送信します。バックエンドの Task Manager はリクエストを受け付け、タスクを生成します。
3.  **タスクの計画と実行:** Task Manager はタスクの実行計画を立て、Agent Core にタスクの実行を指示します。Agent Core は、プロンプト生成、LLM API 呼び出し、Tool Executor を利用したツール実行などを順次行います。
4.  **結果の収集と処理:** Tool Executor はツールの実行結果を Agent Core に返します。Agent Core は結果を解析し、必要に応じて次のアクションを決定します。
5.  **ユーザーへの結果提示:** Agent Core はタスクの実行結果をフロントエンドに送信します。フロントエンドは結果を UI 上に表示し、ユーザーにフィードバックを求めます。
6.  **フィードバックの反映とタスクの継続 (または完了):** ユーザーからのフィードバックに基づき、タスクを継続するか、完了とします。タスクが継続される場合は、ステップ 3 に戻り、次のアクションを実行します。

### 3.2 VS Code 拡張機能版からの主な変更点

#### 3.2.1 ユーザーインターフェース (Web ブラウザベース)

VS Code 拡張機能版では VS Code のサイドバーに UI が表示されていましたが、Web アプリケーション版では Web ブラウザ上で独立した UI を提供します。これにより、VS Code をインストールしていない環境でも Roo-Cline を利用できるようになります。

#### 3.2.2 ファイルシステムアクセス、コマンド実行などのツール実行方法

VS Code 拡張機能版では VS Code の API を利用してファイルシステムへのアクセスやコマンド実行を行っていましたが、Web アプリケーション版ではバックエンドサーバーを介してこれらの操作を行います。

- **ファイルシステムアクセス:** フロントエンドからのファイル操作リクエストはバックエンドサーバーに送信され、サーバー側でファイルシステムを操作します。必要に応じて、ファイルの内容をフロントエンドに送信したり、フロントエンドから送信された内容をファイルに書き込んだりします。
- **コマンド実行:** フロントエンドからのコマンド実行リクエストはバックエンドサーバーに送信され、サーバー側でコマンドを実行します。コマンドの実行結果はフロントエンドに WebSocket などでストリーミング配信したり、完了後にまとめて送信したりします。

#### 3.2.3 状態管理、コンテキストの扱い

VS Code 拡張機能版では VS Code のワークスペースやエディタの状態をコンテキストとして利用していましたが、Web アプリケーション版では、より汎用的な方法で状態管理とコンテキストの扱いを行います。

- **状態管理:** フロントエンド、バックエンドそれぞれで状態管理を行い、必要に応じて状態を同期させます。フロントエンドの状態管理には React の Context API や状態管理ライブラリを利用し、バックエンドの状態管理にはメモリ上の変数、データベース、Redis などのデータストアを利用します。
- **コンテキストの扱い:** タスクの実行に必要なコンテキスト情報（プロジェクトのファイル構造、Git の状態、ユーザーの過去の入力など）をバックエンドで収集・管理し、Agent Core に提供します。

## 4. 修正方針とガイドライン

### 4.1 修正の原則

Roo-Cline Web アプリケーション版の修正においては、以下の原則を重視します。

- **保守性:** 長期的な運用を見据え、コードの可読性、理解しやすさ、変更しやすさを重視します。
- **拡張性:** 新機能の追加や既存機能の変更が容易に行えるよう、柔軟なアーキテクチャとモジュール設計を心がけます。
- **安定性:** 安定した動作を保証するため、十分なテストを行い、エラーハンドリングを適切に実装します。

#### 4.1.1 明確な目的と影響範囲の特定

修正を行う前に、修正の目的、修正によって解決したい課題、期待される効果を明確に定義します。また、修正がシステム全体に及ぼす影響範囲を事前に評価し、予期せぬ副作用を防ぐように努めます。

#### 4.1.2 テスト駆動開発 (TDD) の実践

可能な限り、テスト駆動開発 (TDD) の手法を取り入れ、テストコードを先に記述してから実装コードを書くように心がけます。これにより、設計の初期段階からテスト容易性を考慮し、品質の高いコードを効率的に開発できます。

#### 4.1.3 コードレビューの実施

全てのコード修正に対して、必ずコードレビューを実施します。コードレビューを通じて、コードの品質向上、バグの早期発見、チームメンバー間の知識共有を図ります。

### 4.2 ディレクトリ構成とモジュール分割

プロジェクトのディレクトリ構成は、機能ごとのモジュール分割を基本とし、関心事の分離 (SoC) の原則を適用します。各ディレクトリの役割と責任範囲を明確にし、モジュール間の依存関係を適切に管理することで、保守性と拡張性を高めます。

- **`frontend/`:** フロントエンドアプリケーションのコードを格納します。
    - `components/`: React コンポーネント
    - `contexts/`: React Context
    - `services/`: フロントエンドサービス
    - `utils/`: ユーティリティ関数
    - `assets/`: 静的アセット
- **`backend/`:** バックエンドアプリケーションのコードを格納します。
    - `controllers/`: API コントローラー
    - `services/`: バックエンドサービス (Agent Core, Task Manager, Tool Executor など)
    - `models/`: データモデル
    - `repositories/`: データアクセスロジック
    - `utils/`: ユーティリティ関数
- **`database/`:** データベース関連のファイル (マイグレーションスクリプト、スキーマ定義など)
- **`config/`:** 設定ファイル (環境変数定義、アプリケーション設定など)
- **`docs/`:** ドキュメント (API ドキュメント、アーキテクチャ設計書など)
- **`tests/`:** テストコード (ユニットテスト、結合テスト、E2E テスト)

### 4.3 コーディング規約

プロジェクト全体で一貫したコーディングスタイルを維持するため、以下のコーディング規約を遵守します。

- **言語ごとのコーディングスタイル:** 各言語 (TypeScript, JavaScript, Python など) において、一般的に推奨されるコーディングスタイル (例: ESLint, Prettier, PEP 8) に準拠します。
- **命名規則:** 変数名、関数名、クラス名、ファイル名などは、意味が明確で一貫性のある命名規則に従います。
- **コメント記述:** コードの意図、複雑なロジック、注意点などをコメントとして適切に記述します。特に、API ドキュメント、関数ドキュメント、クラスドキュメントは JSDoc, Sphinx などのツールを用いて自動生成できるように記述します。
- **エラーハンドリング:** エラーが発生する可能性のある箇所では、try-catch ブロックなどで適切にエラーハンドリングを行い、エラーメッセージをログに出力します。
- **ロギング:** アプリケーションの動作状況を把握するため、適切なロギングレベル (debug, info, warn, error) でログを出力します。

### 4.4 API 設計

バックエンド API は RESTful API の原則に基づいて設計します。

- **RESTful なエンドポイント設計:** リソースをベースにしたエンドポイント設計 (例: `/tasks`, `/users`) を行い、HTTP メソッド (GET, POST, PUT, DELETE) を適切に使い分けます。
- **API ドキュメントの整備:** OpenAPI (Swagger) などのツールを用いて API ドキュメントを自動生成し、常に最新の状態に保ちます。
- **バージョン管理:** API の変更を行う場合は、API バージョン (例: `/api/v1/tasks`) を導入し、互換性を維持しながら段階的なアップデートを可能にします。

### 4.5 テスト

品質保証のため、以下の種類のテストを適切に実施します。

- **ユニットテスト:** 各コンポーネント、関数、クラスなどの単体テストを Jest, Mocha, pytest, unittest などのテストフレームワークを用いて記述します。
- **結合テスト:** 複数のコンポーネント間の連携をテストします。
- **E2E テスト:** フロントエンドからバックエンド、データベースまでを含めたシステム全体の結合テストを Cypress, Playwright などの E2E テストフレームワークを用いて記述します。
- **テストカバレッジの目標:** ユニットテスト、結合テストにおいて、一定以上のテストカバレッジ (例: 80% 以上) を目標とします。
- **CI/CD パイプラインへのテスト組み込み:** コードの変更が push されるたびに、CI/CD パイプラインで自動的にテストを実行し、テストにパスした場合のみデプロイを行うようにします。

### 4.6 ドキュメンテーション

以下のドキュメントを整備し、プロジェクトの理解を助け、開発効率を高めます。

- **コードドキュメント:** JSDoc, Sphinx などのツールを用いてコードドキュメントを自動生成します。
- **API ドキュメント:** OpenAPI (Swagger) などのツールを用いて API ドキュメントを自動生成します。
- **アーキテクチャ設計書:** システム全体のアーキテクチャ、コンポーネント構成、データフローなどを記述したドキュメントを作成します。
- **仕様書:** 各機能の仕様、ユースケース、画面遷移などを記述したドキュメントを作成します。
- **変更履歴 (CHANGELOG.md):** バージョンごとの変更点、修正内容、リリース日などを CHANGELOG.md ファイルに記録します。

### 4.7 バージョン管理

Git を用いてソースコードのバージョン管理を行います。

- **Git ワークフロー:** GitHub Flow, Gitflow などの Git ワークフローを採用し、チーム開発におけるブランチ運用、リリース戦略を明確にします。
- **ブランチ戦略:** `main` ブランチ (または `master` ブランチ) を常にリリース可能な状態に保ち、開発は `develop` ブランチ (または feature branches) で行い、Pull Request を通じて `main` ブランチにマージします。
- **コミットメッセージ規約:** Conventional Commits などのコミットメッセージ規約に従い、コミットメッセージを構造化し、変更履歴を追跡しやすくします。
- **タグ付け、リリースプロセス:** リリース時にはバージョン番号のタグを Git リポジトリに付与し、リリースプロセスを自動化します。

## 5. 触ってはいけないファイル

以下のファイルは、システム全体の動作に重要な影響を与える設定ファイルや環境変数ファイルであるため、**原則として直接編集を禁止** します。これらのファイルを変更する必要がある場合は、事前にチーム内で十分な議論を行い、影響範囲を評価した上で、慎重に変更作業を行ってください。

- `.env` ファイル (環境変数ファイル)
- `config/*.json` ファイル (アプリケーション設定ファイル)
- `database/migrations/` ディレクトリ (データベースマイグレーションスクリプト)
- `infra/` ディレクトリ (インフラ構成ファイル)

これらのファイルを誤って変更した場合、アプリケーションが正常に動作しなくなる可能性があります。変更を行う際は、必ずバックアップを取得し、変更内容を Git で管理してください。
