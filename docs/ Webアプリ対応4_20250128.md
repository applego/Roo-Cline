以下では、今回のメッセージ型定義やWebアプリ化に伴う修正・追加実装の現状と、今後取り組むべき残タスクについて整理し、ドキュメントとしてまとめます。

---

## 1. **最新版の現状**

### 1-1. Webアプリ化に伴うメッセージ定義の更新

- **目的:**
  VS Code拡張機能としての実装に依存していたメッセージ定義を、Webアプリケーションとしても再利用しやすい形に統合すること。

- **実装概要:**

    1. **`ExtensionMessage` (拡張機能との通信向け) と `WebviewMessage` (Webアプリ版) の型** を整理し、**`StandaloneMessage`** や **`EnhancedWebviewMessage`** など、スタンドアロンモード（Webアプリ）用の型を追加。
    2. **`webview-ui/src/types/messages.ts`** にて、下記の型を再定義・追加:
        - **`StandaloneMessageType`:** 既存の拡張機能メッセージのtypeに加え、`upsertApiConfiguration` などスタンドアロン用のtypeを追加。
        - **`StandaloneMessage`:** スタンドアロンモードで送受信するメッセージの構造を定義。
        - **`EnhancedWebviewMessage`:** 受信メッセージをWebアプリ内部で扱いやすくするための拡張型。
    3. **`webview-ui/src/services/api/types.ts`** にある `MessageCallback` や `CommunicationHandler` の定義を修正し、
        - 送信は `StandaloneMessage`
        - 受信は `EnhancedWebviewMessage`
          を扱うように変更。
    4. **`webview-ui/src/services/api/websocket-handler.ts`** (WebSocketHandler) を修正し、受信時に `convertToEnhancedWebviewMessage` を用いて型を変換。送信時に `StandaloneMessage` をJSON化して送る実装に統一。
    5. **`webview-ui/src/context/StandaloneStateContext.tsx`** では、`upsertApiConfiguration` などスタンドアロン専用メッセージを送信できるように `sendMessage` の型を `StandaloneMessage` に修正。

- **完了したこと:**
    - **メッセージ型の衝突・型不一致エラー** が概ね解消され、Webアプリケーション版でもVS Code拡張と同等のメッセージ送受信フローを構築できる下地が整った。
    - `state` や `apiConfiguration` など、両環境で扱う可能性があるプロパティを統合的に扱えるようになった。

### 1-2. 設定画面（Settingsなど）・履歴表示のWebアプリ対応

- **目的:**
    - VS Code版 Roo-Cline で提供していた「設定」「履歴表示」「MCP設定」などの機能を、ブラウザ版UIからも操作できるようにする。
- **現状:**
    - **設定画面UI** (Reactでの `SettingsPanel.tsx` など) はすでに概形が実装済み。
    - 設定変更時に送信するメッセージの型が整備され、**「upsertApiConfiguration」メッセージ** でAPI設定を更新する流れが整いつつある。
    - 履歴表示やチャット表示 (`ChatView` など) はMVPレベルで動作。VS Code版とほぼ同等のUIをWebアプリ側でも再現できるよう型・状態管理を修正中。

### 1-3. MCPツール統合への対応

- **目的:**
    - Roo-Cline がサポートする MCP (Model Context Protocol) サーバーを、Webアプリ版でも同様に利用できるようにする。
- **現状:**
    - **`src/services/mcp/McpHub.ts`** や **`server/src/mcp/McpManager.ts`** など、サーバーサイドでのMCP統合ロジックは一部実装済み。
    - WebアプリからMCPツール呼び出し (`use_mcp_tool` など) のメッセージを送受できるように、**`tool-lists.ts`** / **`mode-validator.ts`** との連携も準備済み。
    - ただし、**ブラウザ・サーバー間の最終的な連携**（WebSocketでのMCPリクエスト → サーバー → MCPサーバー呼び出し → レスポンス返却）はまだ完全には仕上がっていない段階。

## 2. **残タスクと優先度**

下記の4つが大きな開発項目として挙げられています。

1. **サーバーサイドのWebSocketハンドラーの実装**

    - 現在、フロント側 (webview-ui) のWebSocketHandlerは整備されつつあるが、サーバーサイド (server 側) のWSハンドラーで細かいルーティング・認証・MCP呼び出しなどをどう扱うか決める必要がある。
    - 履歴や設定をどのタイミングでサーバーが保持・同期するかも要検討。

2. **APIの設定管理機能の実装**

    - `docs/Webアプリ対応3_20250128.md` などでも触れられていたように、APIキーやモデル選択などを永続化し、**ブラウザ再読み込みやサーバー再起動時にも保持** できる仕組みを整備する必要がある。
    - **`server/src/config/ConfigStore.ts`** がバックエンドでの永続化処理を担当。フロントでの「変更 → WebSocket経由でサーバーへ送信 → ファイル/DBに保存 → フロントへ反映」までのフローを仕上げる。

3. **MCPサーバーとの統合**

    - MCPサーバー (`@modelcontextprotocol/sdk` など) との接続管理やツール呼び出し (`use_mcp_tool`) の連携を仕上げる。
    - すでに**`McpManager.ts`** や **`McpHub.ts`** は存在するが、Webアプリ上でのツール実行→レスポンス表示フローを完全に動作させるには、サーバーサイドのWebSocketハンドラーを含めて追加実装が必要。

4. **ファイル操作の実装（read/write/apply_diffなど）**
    - VS CodeのAPIを使っていた箇所（`vscode.workspace` や `TextEdit` など）を、Webアプリ版ではどう扱うか要再検討。
    - 直接サーバーにファイルの読み書きをさせるか、MCP連携で行うかなど、**セキュリティやUXを考慮した設計** が必要。
    - 実装する際は、**`src/core/Cline.ts`** 内のツール呼び出しロジックや **`src/integrations/editor/DiffViewProvider.ts`** 相当の仕組みをサーバーへ移管・拡張することになる。

## 3. **次に着手すべき項目**

上記4項目（サーバーサイドWS、設定管理、MCP連携、ファイル操作）のうち、**どれから実装に着手すべきか** はチームや要件次第ですが、一般的には以下の順を推奨します:

1. **サーバーサイドのWebSocketハンドラー整備**

    - Webアプリ側はすでにWSハンドラーが概ね整備されているため、サーバー側で同様の処理 (メッセージの受理、API設定更新、MCP呼び出しなど) を適切に振り分ける仕組みを先に作ると、他の機能（設定保存やMCP実行）の実装を容易に進められます。
    - フロントとサーバーを双方向にやり取りできる状態にすることで、**「実際に設定を保存する」「ツールを実行してレスポンスを返す」** などを順次拡張できます。

2. **APIの設定管理機能 (ConfigStore) の仕上げ**

    - WebSocketハンドラーが整っていれば、「設定変更をサーバーに送って保存→サーバーがConfigStoreで永続化→フロントに反映」という流れを作りやすいです。
    - 特にAPIキーやモデル設定などはユーザーの利用開始時に必須となるため、優先度は高いです。

3. **MCPサーバーとの統合 (McpManager, McpHub)**

    - ツール呼び出しのやりとりをWebSocket経由で行う際、サーバーサイドでMCPへの接続・コマンド実行結果の受領などを実装し、フロントに返却する形を確立すると良いでしょう。
    - これにより、LLMのツール呼び出し (`use_mcp_tool` など) がWebアプリでもフルに活用可能となります。

4. **ファイル操作 (read_file, write_file, apply_diff など) のWebアプリ移行**
    - 最後に、VS Code API 依存の箇所（ファイル操作やエディタ操作）をどうサーバー経由で行うか検討・実装する流れが自然です。
    - セキュリティを考慮しつつ、サーバー上でファイルを扱うか、別の仕組みを用意するか方針を決める必要があります。

## 4. **今後の全体的な流れ（例）**

1. **WebSocketサーバー (server側) の実装・整備**

    - `server/src/websocket/` などを中心に、メッセージを受信→パース→必要に応じて `ConfigStore.ts` や `McpManager.ts` へ振り分け、処理結果をクライアント(Webアプリ)に返信する仕組みを作る。

2. **設定の永続化 (ConfigStore) ＋ フロントエンドの設定UI連携**

    - Webアプリの設定画面 → 「保存」 → `upsertApiConfiguration` メッセージ送信 → サーバーで `ConfigStore.save()` → 成功後にフロントへACK or 新しい設定を返す
    - これにより、APIキーやモデル選択が確実にサーバー側に保存され、再起動後も維持される。

3. **MCPツール呼び出しフローの完成**

    - LLMが `use_mcp_tool` (or `access_mcp_resource`) を呼んだ際、サーバー側でMCPサーバーに接続・実行し、結果をWebSocketメッセージでフロントへ返す → フロントUIに表示。
    - MCPサーバーが複数存在する場合は、`McpManager.ts` で接続管理を行い、フロントUI (例: `mcpServers`一覧) と同期を取る。

4. **ファイル操作関連のWebアプリ対応**
    - 最後にファイル読込/書込をWebアプリからどのように行うか具体化する。
    - ローカルファイルを扱う場合はサーバーがホスト上のディレクトリを対象に操作する形、またはMCP経由で行う形にするなど、運用方針を固める。

---

## 5. **まとめ**

- **現状:**

    - Webアプリ側のメッセージ型定義がVS Code版と概ね合流し、設定UI・履歴表示・MCP関連のメッセージ送受信が大きく前進した。
    - 型エラーはほぼ解消し、`StandaloneMessage` / `EnhancedWebviewMessage` / `CommunicationHandler` の設計も定まりつつある。

- **残タスク（優先度順）:**
    1. **サーバーサイドのWebSocketハンドラー実装**
    2. **APIの設定管理機能の仕上げ (ConfigStore連携)**
    3. **MCPサーバーとの統合 (McpManager, McpHub)**
    4. **ファイル操作(read/write/apply_diff)をWebアプリで扱う仕組み**

この順序で進めると、フロントエンド（Webアプリ）とサーバーが双方向で安定動作する下地を固めた上で、段階的に高機能ツール呼び出しやファイル操作を実装できるため、開発効率が向上すると考えられます。

以上が最新版の現状と今後の残タスクの概要です。さらなるご質問や優先度の調整が必要であれば、お知らせください。
