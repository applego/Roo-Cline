# Upstreamマージ方針

## 概要

Roo Clineのフォークプロジェクトにおいて、upstreamの変更を安全に取り込むための方針を定義します。

## 1. 現在の実装状況

### Webアプリ対応

- フロントエンド基本UI実装済み
- WebSocket通信基盤実装済み
- ファイル操作機能実装済み
- 設定管理機能一部実装中
- MCPサーバー連携は未着手

## 2. マージ方針

### 2.1 基本方針

1. upstreamの変更は定期的に取り込む（月1回程度）
2. 影響範囲の小さい変更から段階的に取り込む
3. コンフリクトが予想される領域は慎重に確認

### 2.2 優先順位

1. セキュリティアップデート（高優先）
2. バグ修正（中優先）
3. 機能追加・改善（低優先）

### 2.3 対象外とする変更

1. VS Code拡張特有の機能変更
2. Webviewに依存する実装の変更
3. VS Code APIに強く依存する部分

## 3. 実装手順

### 3.1 事前確認

```bash
# upstreamの変更を確認
git remote update upstream
git log --oneline upstream/main ^main
```

### 3.2 変更の取り込み

```bash
# 作業ブランチ作成
git checkout -b merge/upstream-$(date +%Y%m%d)

# upstreamの変更を取り込み
git merge upstream/main
```

### 3.3 コンフリクト解決の優先順位

1. 型定義とインターフェース
2. コアロジック
3. UI実装
4. テストコード

### 3.4 動作確認項目

1. WebSocket通信
2. ファイル操作
3. 設定管理
4. MCPツール連携
5. テストの実行

## 4. 注意点

### 4.1 VS Code依存の置き換え

- `vscode.postMessage` → `WebSocket.send()`
- `acquireVsCodeApi` → カスタム通信インターフェース
- Webview API → Web標準API

### 4.2 設定の互換性維持

- VS Code設定との互換性レイヤーを維持
- 設定の永続化方法の違いに注意

### 4.3 テストの維持

- VS Code固有のモックを置き換え
- Web環境特有のテストケースを追加
